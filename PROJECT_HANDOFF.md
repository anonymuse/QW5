# QW5 Project Handoff

## Project identity

Repository: https://github.com/anonymuse/QW5

QW5 is an independent project. It must not merge Git history with QW3 or QW4.

- QW3 continues as an independent implementation (https://github.com/anonymuse/qw3).
- QW4 remains a frozen architectural artifact (https://github.com/anonymuse/qw4).
- QW5 is the OpenAI/Codex-led implementation.
- QW3, QW4, DS4, llama.cpp, and MLX may be studied as references and baselines.
- Default to clean-room, AI-authored reimplementation after studying reference projects. If direct source adaptation is ever approved, identify it as attributed third-party material and exclude it from claims about original QW5 authorship.

## Project thesis

QW5 will build an AI-authored, evidence-verified, model-specific inference engine for frontier-class Qwen coding and multimodal models on a heterogeneous Apple Silicon cluster.

The project should demonstrate expertise in:

- AI coding and agentic software development.
- Custom LLM inference-engine development.
- Apple Silicon unified-memory optimization.
- Metal and CPU cooperation.
- Sparse mixture-of-experts scheduling.
- Distributed inference over a Thunderbolt 5 mesh.
- Trace-driven model placement.
- Quantization and quality validation.
- Local, offline-capable, provider-independent LLM hosting.
- Reproducible systems research.

QW5 is not intended to be a wrapper around llama.cpp, MLX, or another complete inference runtime.

## GPT-5.6 workflow

Model selection occurs when a Codex task is launched. Do not assume that one task can or should switch among Sol, Terra, and Luna while it is running.

- Use GPT-5.6 Sol for ambiguous architecture, contracts, cross-cutting changes, difficult debugging, integration, benchmark interpretation, polished technical writing, and final review.
- Use GPT-5.6 Terra for separate, bounded implementation tasks after their contracts, owned paths, inputs, and acceptance tests are established.
- Use GPT-5.6 Luna for separate, highly specified and repeatable tasks such as schema generation, fixture construction, mechanical transformations, documentation normalization, and repetitive test expansion.
- Use the lowest reasoning effort that reliably completes the task. Medium is the normal starting point; High or Extra High is appropriate for difficult multi-step work.
- Use Max when a single difficult task benefits from deeper reasoning and unified context.
- Ultra is not a reasoning level. It is a multi-agent execution mode. Use it only when a later task explicitly requests parallel agents and the work can be partitioned into genuinely independent, bounded subtasks.
- Do not use Ultra for the initial repository bootstrap. The foundation, instructions, contracts, and architecture records should be produced coherently by one Sol task.
- After contracts are accepted, Ultra may delegate read-heavy research, independent validation, benchmarks, or work on disjoint owned paths. Sol remains responsible for integration and adversarial review.
- Record the model family and reasoning setting actually used when that information is available. Do not claim a particular model or setting when it cannot be verified.
- Keep durable state in specifications, ADRs, artifacts, provenance records, and the task board—not only in task transcripts.
- Larger context and multi-agent capability reduce coordination cost but do not justify speculative architecture or unverified claims.

## Public positioning

Preferred description:

> QW5 is an AI-authored, human-directed, evidence-verified inference engine for running frontier-class Qwen coding and multimodal agents across a heterogeneous Apple Silicon cluster.

The defensible AI-authorship claim is:

> All original QW5 source code and documentation are generated by frontier AI coding agents. The project is human-directed, publicly reviewed, and empirically verified. Dependencies and cited reference implementations are separately attributed.

Do not imply that model weights, dependencies, Apple frameworks, research, or referenced implementations were AI-generated.

Credibility should come from correctness, transparent measurements, reproducibility, technical decisions, and honest negative results—not code volume or unverified novelty claims.

## Hardware

The initial cluster contains:

- Node A: Apple M5 Pro, 48 GB unified RAM, 1 TB internal SSD.
- Node B: Apple M5 Max, 48 GB unified RAM, 1 TB internal SSD.
- Node C: Apple M5 Max, 48 GB unified RAM, 1 TB internal SSD.
- All nodes run the newest available macOS.
- All three nodes are directly connected in a full Thunderbolt 5 mesh.
- Aggregate physical memory is 144 GB.

Exact GPU configuration, memory bandwidth, macOS build, available SSD space, thermal state, and link characteristics must be collected by QW5 tooling rather than manually assumed.

Node A is the logical control plane but is also available for model computation. Node B and Node C are expected to own more performance-critical work, but placement must be measurement-driven rather than hard-coded.

Future phases may incorporate M4 Mac Minis. They are not part of the initial architecture.

## Model strategy

### First working target

`Qwen/Qwen3-Coder-Next`

Purpose:

- Establish end-to-end correctness.
- Deliver a useful local coding agent early.
- Validate the custom hybrid-attention and MoE implementation.
- Establish single-node and distributed baselines.
- Exercise long-context and tool-calling behavior.

Relevant characteristics:

- 80B total parameters.
- 3B activated parameters.
- 48 layers.
- Hybrid Gated DeltaNet and gated attention.
- 512 experts, 10 activated per token, plus a shared expert.
- Native 262,144-token context.
- Apache-2.0 license.
- Designed for coding agents and local development.

### Flagship target

`Qwen/Qwen3.5-397B-A17B`

Purpose:

- Demonstrate a frontier-class model whose deployment genuinely benefits from the cluster.
- Exercise aggressive mixed quantization, expert placement, direct peer communication, and unified-memory management.
- Begin with text-only inference.
- Add native vision and multimodal operation after text correctness and performance are established.

Relevant characteristics:

- 397B language-model parameters.
- 17B activated parameters.
- 60 layers.
- Hybrid Gated DeltaNet and gated attention.
- 512 experts, 10 routed plus one shared.
- Native 262,144-token context with longer-context extension available.
- Native multimodal architecture.
- Apache-2.0 license.

The theoretical two-bit language-weight floor is approximately 99 GB before quantization metadata, higher-precision tensors, vision components, runtime state, scratch memory, and operating-system headroom. Deployment feasibility must therefore be proven before it is promised.

### Deferred stretch target

`Qwen3-Coder-480B-A35B-Instruct`

This is not the initial flagship because its theoretical two-bit weight floor is approximately 120 GB before runtime overhead. Revisit only after the Qwen3.5 placement and quantization work is mature.

## Runtime boundaries

The preferred implementation is:

- Zig for the host runtime, control logic, memory management, transport, artifact validation, and orchestration.
- Custom Metal kernels for performance-critical GPU operations.
- CPU reference kernels for correctness and selected production operations where measurement shows a benefit.
- Python allowed for offline model inspection, conversion, calibration, oracle generation, and analysis.
- Apple frameworks and focused third-party dependencies are allowed.

QW5 must own:

- Model loading and artifact manifests.
- Tensor layout and validation.
- Decode and prefill execution.
- KV and recurrent state management.
- MoE routing and expert dispatch.
- Quantized kernel execution.
- Distributed scheduling and transport.
- Memory placement and budgeting.
- Runtime telemetry.
- Correctness and benchmark artifacts.

llama.cpp, MLX, Transformers, PyTorch, and vendor runtimes may be used as:

- Correctness oracles.
- Model-conversion references.
- Baselines.
- Offline evaluation tools.
- Behavioral references.

They must not become the QW5 production execution engine.

## Architectural principles

1. Correctness precedes optimization.
2. Model revisions and tensor inventories must be pinned and hashed.
3. Physical measurements precede topology decisions.
4. Node roles are logical defaults, not permanent placement constraints.
5. The M5 Pro participates in compute when beneficial.
6. Thunderbolt transfers should be peer-to-peer when that avoids unnecessary relay.
7. Packets are counted by actual destination and message, not by activated-expert count.
8. Prefill and autoregressive decode require separate performance models.
9. SSD is initially a cold artifact and promotion tier, not a steady-state active-weight assumption.
10. CPU execution is used where measurement shows correctness, overlap, or performance value.
11. Expert placement should incorporate routing frequency, quantization sensitivity, compute throughput, and network cost.
12. Simulations, estimates, targets, and measured results must be clearly distinguished.
13. Negative findings are valid project outcomes and should be published.

## Distributed message classes

Keep distinct protocol semantics for at least:

- Layer or recurrent-state transfer.
- Expert dispatch.
- Expert result.
- Control-plane commands.
- Telemetry and health.
- Artifact and model identity negotiation.

Do not encode the project around a mandatory coordinator relay.

## Benchmark requirements

Every published result must identify:

- QW5 commit.
- Model repository, exact revision, and artifact hashes.
- Quantization scheme by tensor class.
- Node hardware and operating-system build.
- Available and allocated memory.
- Link topology and measured link behavior.
- Context length, prompt length, generation length, and sampling settings.
- Prefill and generation results separately.
- Thermal and power conditions when available.
- Whether the result is measured, simulated, estimated, or targeted.
- Comparable baseline configuration.

No absolute performance promise should be made before the baseline and feasibility phases.

## AI provenance requirements

Each pull request should record:

- Coding tool and model family.
- Task objective.
- Material constraints supplied by the human project owner.
- Files generated or materially changed.
- Tests and benchmarks executed.
- Review model or review process.
- Human decisions or interventions.
- External source or dependency attribution.
- Whether any non-AI-authored source was adapted.

Private reasoning traces are not required. The public record should contain enough information to substantiate the development method without exposing secrets or private conversation history.

## Development workflow

- Start each task from current `main`.
- Use one task, one `codex/<task-name>` branch, one worktree, and one pull request.
- Keep a stable integration checkout on `main`.
- Use Codex-managed worktrees for task work.
- Read `README.md`, `PROJECT_HANDOFF.md`, `AGENTS.md`, and active coordination documents before editing.
- Check repository status before making changes.
- Keep durable project state in repository documents, not only in task transcripts.
- Run relevant validation before committing.
- Commit focused changes.
- Open draft pull requests when changes are ready for review.
- Do not delete branches or worktrees without explicit confirmation.

Use GPT-5.6 Sol for architectural decisions, difficult debugging, and final review; Terra for bounded implementation and test work; and Luna for mechanical or repetitive generation where appropriate.

## Initial public demonstration

The intended showcase is a repo-scale coding task that:

1. Loads substantial repository context.
2. Uses QW5 as an OpenAI-compatible local coding-model endpoint.
3. Continues after public network access is disconnected.
4. Uses tools to inspect, edit, build, and test the repository.
5. Produces a verified patch.
6. Displays live cluster placement, memory, TB5 traffic, power, and inference performance.
7. Publishes the full reproducibility record.

Frame this around local control, resilience, privacy, inspectability, and provider independence. Avoid centering the project on a temporary provider controversy.
